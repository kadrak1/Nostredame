## Фаза 1 — MVP · *Завершена (100%)*

**Цель**: запустить минимальную рабочую систему бронирования.

### Что входит

| Область | Ключевые возможности |
|---------|---------------------|
| **Конструктор зала** | Drag-and-drop расстановка столов, стены, сохранение плана в JSON |
| **Каталог табаков** | CRUD: название, бренд, крепость (1–5), вкусы, наличие |
| **Бронирование** | Выбор даты/времени/стола, ввод контактов, статусы |
| **Админ-панель** | Дашборд броней, подтверждение/отклонение, просмотр зала |
| **Безопасность** | JWT, шифрование телефонов, rate limiting, audit log, HTTPS |

### Критерии выхода из Фазы 1

- [x] Гость проходит полный флоу бронирования через веб
- [x] Админ видит, подтверждает и отклоняет брони
- [x] Все данные передаются только по HTTPS
- [x] Персональные данные зашифрованы в БД
- [x] Audit log ведётся для всех действий администраторов

**Все тикеты Фазы 1 завершены** — 20/20.

## Фаза 3 — Уведомления и PWA · *Запланировано*

**Цель**: информировать гостей о статусах без лишних вопросов персоналу.

### Что входит

| Область | Ключевые возможности |
|---------|---------------------|
| **Telegram-уведомления** | Бронь подтверждена/отменена, напоминание за 2 часа, заказ принят |
| **Web Push** | Те же события через браузерные push-уведомления |
| **PWA** | Service Worker, установка на экран, работа офлайн (просмотр истории) |
| **Управление настройками** | Гость включает/отключает каналы независимо |

## Принципы приоритизации

1. **Сначала бронирование** — core value продукта, без него остальное не нужно
2. **Безопасность с первого дня** — не откладывать шифрование и rate limiting
3. **Mobile-first** — любая новая фича сначала проектируется для мобильного
4. **Один канал за раз** — не запускать новый канал (SMS, Push, Telegram) без завершения предыдущего
