# PRD: Система бронирования и заказа кальянов для кальянных заведений

**Рабочее название**: HookahBook
**Версия**: MVP v1.0
**Дата**: 2026-02-26

---

## Context

Кальянные заведения испытывают типичные операционные проблемы: посетители ждут кальянщика для заказа, бронирование столов происходит по телефону без визуализации зала, постоянные клиенты не могут быстро повторить свой заказ. Система решает эти проблемы через единую платформу бронирования и онлайн-заказа кальянов с поддержкой веб и Telegram. MVP делается под одно конкретное заведение с потенциалом масштабирования. Хостинг — Raspberry Pi 5 на собственном домене.

---

## Problem Statement

Посетители кальянных тратят время на ожидание кальянщика, не могут заранее выбрать и заказать кальян при бронировании, а постоянные клиенты вынуждены каждый раз заново описывать свои предпочтения. Владельцы заведений не имеют удобного инструмента для управления бронированием с визуализацией зала. Это приводит к потере клиентов, медленному обслуживанию и неоптимальной загрузке зала.

---

## Goals

1. **Сократить время от посадки до получения кальяна** — клиент заказывает до или сразу после посадки, без ожидания кальянщика
2. **Повысить конверсию броней** — визуальный выбор стола + предзаказ кальяна снижают барьер бронирования
3. **Увеличить retention постоянных клиентов** — история заказов и быстрый повторный заказ
4. **Дать владельцу контроль над залом** — конструктор зала с визуализацией загрузки столов
5. **Подготовить интеграцию с учётом ресурсов** — API-контракт для будущего сервиса списания табака

---

## Non-Goals (Out of Scope для MVP)

1. **Онлайн-оплата** — в MVP только офлайн-расчёт на месте (нет интеграции с платёжными системами)
2. **Мультитенантность** — система под одно заведение, поддержка нескольких заведений — следующая фаза
3. **Сервис учёта ресурсов** — отдельный проект; MVP только предоставляет API-контракт для интеграции
4. **WhatsApp и другие мессенджеры** — только Telegram + веб в MVP
5. **Программа лояльности / скидки** — не входит в MVP
6. **Заказ еды и напитков** — только кальяны, не входит в MVP

## User Personas

### 1. Гость (посетитель)
- Хочет забронировать стол и предзаказать кальян
- Хочет заказать кальян, сидя за столом, без ожидания
- Постоянный гость хочет повторить прошлый заказ

### 2. Владелец / Администратор заведения
- Настраивает зал (конструктор)
- Управляет меню табаков и их наличием
- Видит текущие и предстоящие бронирования
- Подтверждает/отклоняет брони

### 3. Кальянщик (hookah master)
- Получает заказы с деталями (крепость, табаки, время приготовления)
- Видит очередь заказов

## User Stories

### Гость — Бронирование

- **US-1**: Как гость, я хочу увидеть план зала с доступными столами на нужную дату/время, чтобы выбрать удобное место
- **US-2**: Как гость, я хочу при бронировании предзаказать кальян (крепость, вкусы из наличия, время приготовления), чтобы он был готов к моему приходу
- **US-3**: Как гость, я хочу получить подтверждение брони в Telegram или на веб-странице
- **US-4**: Как гость, я хочу отменить или перенести бронь

### Гость — Заказ за столом

- **US-5**: Как гость за столом, я хочу отсканировать QR-код и выбрать способ взаимодействия (веб или Telegram)
- **US-6**: Как гость за столом, я хочу выбрать крепость кальяна и получить рекомендации табаков из наличия, соответствующих этой крепости
- **US-7**: Как гость за столом, я хочу собрать микс табаков и указать предпочтения по приготовлению
- **US-8**: Как гость за столом, я хочу видеть статус приготовления моего заказа

### Постоянный гость

- **US-9**: Как постоянный гость, я хочу войти по номеру телефона (SMS OTP), чтобы увидеть свою историю
- **US-10**: Как постоянный гость, я хочу повторить предыдущий заказ в один клик
- **US-11**: Как постоянный гость, я хочу видеть свои прошлые заказы и избранные миксы

### Владелец / Администратор

- **US-12**: Как владелец, я хочу создать план зала в визуальном конструкторе (столы, стены, зоны)
- **US-13**: Как владелец, я хочу задать вместимость каждого стола
- **US-14**: Как владелец, я хочу управлять каталогом табаков (название, бренд, крепость, вкус, наличие)
- **US-15**: Как владелец, я хочу видеть дашборд с бронированиями на сегодня и загрузкой зала
- **US-16**: Как владелец, я хочу подтверждать или отклонять входящие брони

### Кальянщик

- **US-17**: Как кальянщик, я хочу видеть очередь заказов с деталями (стол, крепость, табаки, время)
- **US-18**: Как кальянщик, я хочу отмечать статус заказа (принят, готовится, подан)

## API-контракт для сервиса учёта ресурсов (P2, но проектируем сейчас)

Система будет публиковать события через webhook/message queue при:
- Подтверждении бронирования с предзаказом → `booking.confirmed` (список табаков, граммовка)
- Оформлении заказа за столом → `order.created` (список табаков, граммовка)
- Отмене брони → `booking.cancelled` (возврат табаков в доступные)

Формат события (draft):
```json
{
  "event": "order.created",
  "venue_id": "...",
  "order_id": "...",
  "items": [
    { "tobacco_id": "...", "name": "...", "weight_grams": 15 }
  ],
  "timestamp": "..."
}
```

В MVP: события логируются, но не отправляются. Endpoint для подписки на события будет реализован, но без внешних потребителей.

## Success Metrics

### Leading Indicators (первые 2 недели)
- Количество бронирований через систему vs по телефону: цель **>30%** через систему
- Среднее время от посадки до заказа кальяна: цель **< 2 мин** (vs текущие ~5-10 мин)
- % бронирований с предзаказом кальяна: цель **>40%**

### Lagging Indicators (1-2 месяца)
- Доля повторных посетителей, использующих "повторить заказ": цель **>20%**
- Снижение нагрузки на персонал по приёму заказов: качественная оценка от владельца
- NPS от гостей, использовавших систему: цель **>50**

## Архитектура (высокоуровневая для MVP)

### Хостинг
- **Raspberry Pi 5** — сервер приложения + БД
- **Собственный домен** — HTTPS через Let's Encrypt / Cloudflare tunnel

### Рекомендуемый стек (для обсуждения)
- **Backend**: Python (FastAPI) или Node.js (Express/Fastify) — легковесный, хорошо работает на ARM
- **Database**: PostgreSQL или SQLite (для MVP SQLite может быть достаточно)
- **Frontend**: React/Next.js или Vue/Nuxt — SPA с SSR для SEO страницы бронирования
- **Telegram Bot**: python-telegram-bot / grammY (Node.js)
- **Конструктор зала**: Canvas-based (Fabric.js / Konva.js) или SVG-based
- **SMS OTP**: Внешний сервис (Twilio / SMS.ru / аналог)
- **Reverse proxy**: Nginx или Caddy

### Схема данных (ключевые сущности)

```
Venue          → floor_plan (JSON), settings
Table          → venue_id, number, capacity, x, y, width, height
Tobacco        → venue_id, name, brand, strength (1-5), flavor_profile, in_stock
Booking        → venue_id, table_id, guest_id?, guest_phone, date, time_from, time_to, status, guest_count
HookahOrder    → booking_id?, table_id, guest_id?, strength, prep_time, status
OrderItem      → order_id, tobacco_id, weight_grams
Guest          → phone, name?, telegram_id?, created_at
OrderHistory   → guest_id, order_id (для повторных заказов)
User (admin)   → venue_id, role (owner/hookah_master), login, password_hash
```

## Timeline Considerations

- **Фаза 1 (MVP Core)**: Конструктор зала + бронирование + каталог табаков + веб-интерфейс гостя + админ-панель
- **Фаза 2 (MVP Complete)**: Telegram-бот + заказ за столом (QR) + панель кальянщика + SMS OTP
- **Фаза 3 (Post-MVP)**: История заказов + повторный заказ + уведомления + API для учёта ресурсов

Жёсткие зависимости:
- Конструктор зала нужен **до** бронирования (нет плана → нечего показывать гостям)
- Каталог табаков нужен **до** заказа кальянов
- SMS OTP требует выбора и настройки провайдера

## Verification

После реализации каждой фазы:
1. **Конструктор зала**: владелец создаёт план реального зала, столы отображаются корректно на стороне гостя
2. **Бронирование**: гость бронирует стол на конкретное время, второй гость не может забронировать тот же стол на то же время
3. **Заказ кальяна**: гость выбирает крепость → видит отфильтрованные табаки → создаёт заказ → кальянщик видит заказ в очереди
4. **QR-код**: сканирование ведёт на страницу выбора канала, deep link в Telegram корректно привязывает к столу
5. **Telegram-бот**: полный флоу бронирования через бота
6. **Нагрузка на RPi5**: проверить производительность при 10+ одновременных пользователях
