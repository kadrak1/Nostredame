## Эпик 0: Инфраструктура и безопасность

### T-001: ✅ Инициализация проекта
**Тип**: setup
**Описание**: Создать структуру монорепозитория, настроить Docker
- Создать `backend/`, `frontend/`, `docker-compose.yml`, `Caddyfile`
- Dockerfile для backend (Python 3.12) и frontend (Node 20)
- docker-compose с сервисами: backend, frontend (dev), caddy
- `.env.example` с описанием всех переменных (без реальных значений)
- `.gitignore` (исключить `.env`, `*.db`, `node_modules`, `__pycache__`)
- **AC**: `docker-compose up` поднимает все сервисы, API отвечает на `/health`

### T-002: ✅ Настройка FastAPI + SQLite + SQLAlchemy
**Тип**: backend
**Описание**: Базовая конфигурация backend
- FastAPI app с CORS middleware
- SQLAlchemy async engine + SQLite
- Alembic для миграций
- Конфиг через Pydantic Settings (из `.env`)
- Health check endpoint `GET /api/health`
- **AC**: `alembic upgrade head` создаёт БД, `/api/health` возвращает `200`

### T-003: ✅ Безопасность — фундамент
**Тип**: security
**Описание**: Базовые меры безопасности с первого дня
- **HTTPS only** — Caddy настроен с автоматическим редиректом HTTP → HTTPS
- **CORS** — разрешены только конкретные origins (домен, localhost для dev)
- **Rate limiting** — `slowapi` на критичных endpoints (auth, бронирование): 10 req/min на IP для auth, 30 req/min для остальных
- **Helmet-аналог** — security headers через Caddy: `X-Content-Type-Options`, `X-Frame-Options`, `Strict-Transport-Security`, `Content-Security-Policy`
- **Input validation** — все входные данные через Pydantic-схемы (автоматическая защита от injection)
- **SQL injection** — SQLAlchemy parametrized queries (по умолчанию безопасно, но добавить линтер-проверку на raw SQL)
- **Файл БД** — права доступа `600` (только владелец), расположение вне webroot
- **AC**: Все security headers присутствуют в ответах, rate limit срабатывает при превышении

### T-004: ✅ Безопасность — хранение секретов и данных
**Тип**: security
**Описание**: Защита персональных данных и секретов
- **Переменные окружения** — все секреты (JWT secret, SMS API key, Telegram token) только через `.env`, никогда в коде
- **JWT secret** — минимум 256 бит, генерация через `secrets.token_urlsafe(32)`
- **Телефоны в БД** — хранить в хэшированном виде (для поиска) + зашифрованном (для отображения). Использовать `cryptography.fernet` с ключом из `.env`
- **OTP коды** — хранить хэш (bcrypt), TTL 5 минут, максимум 3 попытки, после чего блокировка на 15 минут
- **Логирование** — никогда не логировать телефоны, OTP, токены. Маскировать PII в логах (`+7***1234`)
- **Бэкапы БД** — cron-скрипт для ежедневного бэкапа SQLite файла (cp + timestamp), хранить последние 7 дней
- **AC**: grep по коду не находит hardcoded секретов, телефоны в БД зашифрованы

### T-005: ✅ Настройка React + Vite
**Тип**: frontend
**Описание**: Инициализация frontend проекта
- React 18 + TypeScript + Vite
- React Router v6 (маршрутизация)
- Axios + React Query (API-клиент с кешированием)
- Базовый layout: навигация, роутинг страниц
- Proxy API-запросов на backend (vite.config.ts)
- **XSS-защита**: React по умолчанию экранирует вывод; добавить ESLint правило `no-dangerously-set-inner-html`
- **AC**: Фронтенд открывается, API-запросы проксируются на backend

## Эпик 1: Управление заведением (админ)

### T-010: ✅ Модели данных — Venue, Table, User (admin)
**Тип**: backend
**Описание**: ORM-модели и миграции
- `Venue`: id, name, address, phone, working_hours (JSON), floor_plan (JSON), created_at
- `Table`: id, venue_id (FK), number, capacity, x, y, width, height, shape (rect/circle), is_active
- `User`: id, venue_id (FK), role (owner/admin/hookah_master), login, password_hash, created_at
- **Пароли** — bcrypt через `passlib[bcrypt]`, минимум 8 символов
- Alembic миграция
- **AC**: Миграция применяется без ошибок, модели создают таблицы

### T-011: ✅ Аутентификация админов
**Тип**: backend
**Описание**: JWT-аутентификация для админ-панели
- `POST /api/auth/login` — login + password → JWT access + refresh token
- `POST /api/auth/refresh` — обновление access token
- JWT access token: TTL 30 минут, в заголовке Authorization
- JWT refresh token: TTL 7 дней, httpOnly cookie (secure, SameSite=Strict)
- Middleware для проверки ролей (owner, admin, hookah_master)
- **Brute-force защита**: блокировка на 15 мин после 5 неудачных попыток входа (по IP + login)
- **AC**: Логин возвращает токены, защищённые endpoints отклоняют запросы без токена (401)

### T-012: ✅ CRUD заведения
**Тип**: backend
**Описание**: API для управления заведением
- `GET /api/venue` — данные заведения (публичный, без чувствительных полей)
- `PUT /api/venue` — обновление (только owner)
- Рабочие часы: JSON `{ "mon": {"open": "12:00", "close": "02:00"}, ... }`
- **AC**: Owner может обновить данные, неавторизованный — только читать публичные поля

### T-013: ✅ API конструктора зала
**Тип**: backend
**Описание**: Сохранение и загрузка плана зала
- `GET /api/venue/floor-plan` — текущий план зала (JSON)
- `PUT /api/venue/floor-plan` — сохранение плана (только owner/admin)
- `GET /api/tables` — список столов с координатами и вместимостью
- `POST /api/tables` — добавить стол
- `PUT /api/tables/{id}` — обновить стол
- `DELETE /api/tables/{id}` — удалить стол (soft delete: is_active=false)
- Валидация: номер стола уникален, capacity > 0, координаты в пределах зала
- **AC**: План сохраняется и загружается, валидация работает

### T-014: ✅ Фронтенд — конструктор зала (react-konva)
**Тип**: frontend
**Описание**: Визуальный редактор плана зала для владельца
- Canvas на react-konva с сеткой
- Добавление столов: drag из палитры на canvas
- Формы столов: круг, прямоугольник
- Изменение размера и позиции (drag-and-drop)
- Настройка стола: номер, вместимость (popup при клике)
- Добавление стен/перегородок (линии/прямоугольники)
- Кнопки: сохранить, отменить изменения
- Сохранение в JSON → `PUT /api/venue/floor-plan`
- **AC**: Владелец может расставить столы и стены, сохранить, перезагрузить — план на месте

### T-015: ✅ Фронтенд — админ-панель (каркас)
**Тип**: frontend
**Описание**: Базовый layout админ-панели
- Страница логина
- Sidebar навигация: Зал, Табаки, Бронирования, Заказы
- Защищённые маршруты (redirect на логин без токена)
- **Безопасность**: JWT хранить в httpOnly cookie (не в localStorage — защита от XSS)
- Auto-logout при истечении refresh token
- **AC**: Логин работает, навигация между разделами, logout

## Эпик 3: Бронирование

### T-030: ✅ Модели данных — Booking, Guest
**Тип**: backend
**Описание**: ORM-модели бронирования и гостя
- `Guest`: id, phone_hash (для поиска), phone_encrypted (для отображения), name, telegram_id, created_at
- `Booking`: id, venue_id (FK), table_id (FK), guest_id (FK, nullable), guest_phone_encrypted, date, time_from, time_to, guest_count, status (pending/confirmed/cancelled/completed), notes, created_at, updated_at
- Индекс на (table_id, date, time_from, time_to) для быстрой проверки конфликтов
- **AC**: Модели и миграции работают

### T-031: ⬜ API бронирования
**Тип**: backend
**Описание**: Endpoints для создания и управления бронированиями
- `GET /api/bookings/available-tables?date=...&time_from=...&time_to=...&guests=...` — свободные столы на дату/время (публичный)
- `POST /api/bookings` — создать бронь (публичный, требует телефон)
- `GET /api/bookings/{id}` — статус брони (по ID + телефон или токен)
- `PUT /api/bookings/{id}/cancel` — отменить бронь
- `GET /api/admin/bookings?date=...` — все брони на дату (только admin)
- `PUT /api/admin/bookings/{id}/confirm` — подтвердить (только admin)
- `PUT /api/admin/bookings/{id}/reject` — отклонить (только admin)
- **Проверка конфликтов**: при создании брони проверить, что стол свободен (SELECT ... WHERE overlaps)
- **Защита от спама**: rate limit 5 броней/час с одного IP, captcha при подозрении
- **AC**: Бронь создаётся, конфликты отклоняются, админ подтверждает/отклоняет

### T-032: ⬜ Фронтенд — выбор стола и бронирование (гость)
**Тип**: frontend
**Описание**: Страница бронирования для гостей (mobile-first)
- Шаг 1: Выбор даты, времени, количества гостей
- Шаг 2: Визуализация зала (react-konva, read-only) — свободные столы зелёные, занятые серые
- Шаг 3: Выбор стола (клик → информация о вместимости)
- Шаг 4: Ввод данных (имя, телефон)
- Шаг 5: Подтверждение — показать сводку, кнопка "Забронировать"
- Экран успеха: номер брони, детали, кнопка "Добавить кальян" (заглушка для Фазы 2)
- **AC**: Гость проходит весь флоу, бронь создаётся, недоступные столы нельзя выбрать

### T-033: ⬜ Фронтенд — управление бронированиями (админ)
**Тип**: frontend
**Описание**: Дашборд бронирований в админ-панели
- Календарь / список броней на выбранную дату
- Карточка брони: стол, время, гости, статус, контакт
- Кнопки: подтвердить, отклонить, отметить как завершённую
- Визуализация зала с цветовой индикацией загрузки (сколько столов занято в выбранное время)
- **AC**: Админ видит брони на сегодня, может подтвердить/отклонить

## Эпик 4: Безопасность — аудит и мониторинг

### T-040: ⬜ Логирование и аудит
**Тип**: security
**Описание**: Безопасное логирование действий
- Structured logging (JSON формат) через `structlog`
- Audit log: все действия админов (логин, изменение зала, подтверждение брони) в отдельную таблицу `AuditLog(user_id, action, details, ip, timestamp)`
- **PII masking**: middleware для автоматической маскировки телефонов и токенов в логах
- Ротация логов: logrotate, хранить 30 дней
- **AC**: Все админ-действия записываются в audit log, grep по логам не находит открытых телефонов

### T-041: ⬜ Защита RPi5
**Тип**: devops/security
**Описание**: Базовая защита сервера
- Docker containers с non-root пользователем
- Firewall (ufw): открыты только порты 80, 443, 22 (SSH)
- SSH: только по ключу, отключить парольный вход
- fail2ban для защиты SSH от brute-force
- Автоматические обновления безопасности (unattended-upgrades)
- **AC**: `nmap` снаружи видит только 80, 443, 22; парольный SSH отклоняется
